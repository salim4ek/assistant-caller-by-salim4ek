# Тестирование

Как проверять NN+ Ассистент-Вызов: принципы, тестовая среда, сценарии по ролям, чек-лист перед релизом и формат баг-репорта. Документ для ручных тестировщиков и для смоук-проверок API.

## Принципы

- **Никогда не тестируем на проде с реальными данными.** В системе циркулируют ФИО сотрудников и текст вызовов — это персональные данные. Нагрузочные и деструктивные тесты, рассылки и «звонки всем» — только на отдельном экземпляре.
- **Изолированная среда.** Поднимаем чистый экземпляр с пустой базой и демо-данными (вымышленные ФИО). См. «Тестовая среда».
- **Реальные устройства для push.** Web-push и звук невозможно полноценно проверить в эмуляторе — нужны реальные Android, iPhone и Windows-десктоп.
- **Проверяем поведение, а не «на глаз».** Каждый сценарий = предусловие → действие → ожидаемый результат.
- **Негативные кейсы обязательны.** Не только «работает», но и «корректно отказывает» (чужой токен, неверный пароль, офлайн, гонки).
- **После прохождения — чистим тестовые данные.**

## Тестовая среда

Отдельный экземпляр рядом с продом, своя база и порт, наружу не торчит:

```bash
# своя БД-вольюм + свой порт, тот же образ
docker compose -p accaller_test up -d   # либо отдельный compose с APP_PORT=8090 и чистым volume
# демо-админ задаётся через ADMIN_USERNAME/ADMIN_PASSWORD в .env тестового экземпляра
```

Демо-данные (вымышленные): админ `admin`, врачи «Иванов И.И.», «Петрова А.А.», ассистенты «Кабинет 12», «Кабинет 7». Реальные ФИО в тестах не использовать.

## Матрица: роль × функция

| Функция | Врач | Ассистент | Админ |
|---|---|---|---|
| Регистрация / вход | ✓ (после одобрения) | ✓ (ID или ФИО) | ✓ (логин/пароль) |
| Вызов ассистента | ✓ | — | ✓ (test-call) |
| Принять/отклонить вызов | — | ✓ | — |
| Сообщение второй стороне | ✓→ассистенту видит ответ | ✓→врачу | рассылка всем |
| Одобрение врачей | — | — | ✓ |
| Онлайн-присутствие | виден админу | виден врачу и админу | видит всех |
| Push при закрытом приложении | ✓ | ✓ | — |

## Сценарии

### 1. Регистрация и одобрение врача
1. Врач регистрируется (ФИО + пароль ≥4) → «Заявка отправлена», войти ещё нельзя.
2. На экране ожидания нажать «Уведомить, когда одобрят» → разрешить уведомления (на iPhone — из установленного PWA).
3. Админ в панели «Заявки на регистрацию» → «Подтвердить».
4. **Ожидаемо:** врачу приходит push «✅ Заявка одобрена» (даже при закрытом приложении); если вкладка открыта — автоматический вход.
5. Негатив: вход до одобрения → отказ; «Отклонить» → войти нельзя.

### 2. Аутентификация
- Врач: верные ФИО+пароль → вход; неверные → понятная ошибка; чужая роль на чужом разделе → редирект.
- Ассистент: вход по `NN-XXXXXX` и по ФИО/кабинету; несколько тёзок → просьба войти по ID.
- Админ: верный/неверный логин-пароль.
- Токен живёт 30 дней; протухший/битый токен не пускает на защищённые разделы.

### 3. Вызов
- **Принятие:** врач выбирает ассистента, пишет текст (напр. «кабинет 12»), «Вызвать» → у ассистента полноэкранное окно + звук + системное уведомление; «Принять» → врач видит «Принято: <имя>», звук стихает.
- **Отклонение:** ассистент «Отклонить» с причиной → врач видит причину.
- **Отмена:** врач «Отменить» до ответа → у ассистента вызов исчезает.
- **Таймаут:** никто не ответил за заданное время → вызов снимается сам, врач видит сообщение.
- **Офлайн-ассистент:** приложение закрыто → приходит web-push; при открытии «зависший» вызов до-доставляется.
- Гонка: одновременные «Принять» и «Отменить» — состояние согласованное, без зависших вызовов.

### 4. Сообщения и рассылка
- Ассистент → врач: короткое сообщение, врач получает окно + звук + (если закрыт) push.
- Админ → рассылка всем онлайн: получают и врачи, и ассистенты.

### 5. Web-push (реальные устройства)
- **Android (Chrome/PWA):** разрешить уведомления → вызов при закрытом приложении приходит. Если молчит — снять ограничение батареи для приложения (Настройки → Приложения → Батарея → «Без ограничений»), не смахивать из недавних.
- **iPhone:** только установленный на «Экран Домой» PWA, iOS ≥16.4; разрешить уведомления внутри PWA; телефон не в «Без звука»/Фокусе. Свой звук при закрытом приложении невозможен — играет системный звук уведомления.
- **Windows-десктоп:** окно поверх всех при вызове, трей, автозапуск.

### 6. Присутствие (presence)
- Ассистент вошёл/вышел → у врача и в админке статус online/offline меняется ≤ нескольких секунд.
- Врач вошёл → появляется в админке в «Врачи онлайн».

### 7. Безопасность
- Защищённые роуты требуют корректный JWT нужной роли (чужая роль → 403/редирект).
- Параметризованный SQL (инъекции в ФИО/сообщение не ломают выборку).
- Валидация ввода: пустые/слишком короткие поля → понятный отказ.
- Подписка ожидающего врача на push — только по верным ФИО+паролю (без выдачи прав до одобрения).

## Чек-лист перед релизом

- [ ] Все три роли логинятся; негативные кейсы корректно отказывают.
- [ ] Полный цикл вызова: принять / отклонить / отменить / таймаут.
- [ ] Push приходит при закрытом приложении на Android, iPhone, Windows.
- [ ] Одобрение врача → push об одобрении + авто-вход.
- [ ] Presence (онлайн-ассистенты и онлайн-врачи) корректен.
- [ ] Рассылка админа доходит всем онлайн.
- [ ] Вёрстка на iPhone: контент не под «островом»; ключевые кнопки крупные.
- [ ] Реконнект WebSocket после потери сети/возврата на вкладку.
- [ ] Нет ошибок в консоли браузера на всех экранах.

## Смоук-проверки API

Быстрые автоматические проверки бэкенда (без UI) — на тестовом экземпляре:

- `POST /admin/login`, `/auth/doctor/login`, `/auth/assistant/login` — позитив/негатив.
- `POST /auth/doctor/register` → `POST /auth/doctor/pending-subscribe` (верные/неверные креды) → `POST /admin/doctors/{id}/approve` → факт отправки push врачу.
- `GET /admin/online` — поля `online` и `doctors_online`.
- `GET /auth/vapid-public` (публичный), `GET /healthz`.
- Доставку web-push на конкретную подписку проверяем библиотекой `pywebpush` по сохранённой подписке: статус `201` = принято push-сервисом (FCM/Apple/WNS).

После прогона — удалить тестовые аккаунты.

## Баг-репорт

Минимум: роль и устройство/браузер, шаги воспроизведения, ожидаемое и фактическое поведение, скриншот/запись, время (для сопоставления с логами `docker logs assistant-caller`). Помечать критичность: блокер / серьёзный / косметика.
