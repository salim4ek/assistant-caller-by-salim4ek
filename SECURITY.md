# Безопасность и секреты

Документ о том, как безопасно развернуть проект и что в нём уже сделано для защиты.

---

## Перед деплоем — обязательно

Репозиторий поставляется с **безопасными заглушками** в `.env.example`. Для продакшена их нужно переопределить в собственном `.env`.

### 1. JWT-секрет

```bash
openssl rand -base64 48
```

Впишите результат в `JWT_SECRET`. Слабый или угаданный секрет = возможность подделать токены аутентификации. В проекте секрет должен быть длинным и случайным (≥ 48 байт).

### 2. Пароль первого админа

Первый админ создаётся автоматически из `ADMIN_USERNAME` + `ADMIN_PASSWORD` при пустой базе. Задайте сильные значения:

```env
ADMIN_USERNAME=ваш_админ
ADMIN_PASSWORD=<длинный-надёжный-пароль>
```

После первого входа создайте отдельных админов в панели и смените/отзовите сид-админа.

### 3. VAPID-ключи (web-push)

```bash
npx web-push generate-vapid-keys
```

Впишите `VAPID_PUBLIC` / `VAPID_PRIVATE` в `.env`. Приватный ключ — секрет, в репозиторий не коммитится.

## Что уже защищено в коде

| Механизм | Реализация |
|----------|-----------|
| **JWT** | HS256 с **запинненым методом подписи** — атаки `alg:none` и RS↔HS confusion невозможны; проверяется подпись и срок |
| **Пароли** | `bcrypt` (необратимый хеш + соль) |
| **RBAC** | Каждая группа роутов закрыта middleware по роли: `/admin/*` → только `admin` и т.д. |
| **SQL** | Только параметризованные запросы — защита от инъекций |
| **WebSocket** | Требует валидный JWT в handshake |
| **Security headers** | HSTS, `X-Frame-Options: DENY`, `X-Content-Type-Options: nosniff`, `Referrer-Policy`, `Permissions-Policy` (на reverse-proxy) |
| **Attack surface** | Приложение слушает только `127.0.0.1` за Caddy/TLS; наружу открыт лишь 443 |

## Чего НЕТ в репозитории

В `.gitignore` исключено всё чувствительное:

- реальные `.env` (только `.env.example` с заглушками);
- JWT- и VAPID-ключи, любые `*.key` / `*.pem` / `*.crt`;
- файлы базы SQLite (`*.db`, WAL/SHM);
- серверные бинарники и сборки фронтенда;
- `node_modules`, артефакты PyInstaller;
- IP-адреса, домены и пароли инфраструктуры;
- персональные данные пользователей (вся БД исключена).

## Рекомендации для продакшена

1. Сильный `JWT_SECRET` (`openssl rand -base64 48`) и длинный пароль админа.
2. Reverse-proxy с TLS (Caddy / nginx / Traefik) — **не** выставляйте порт приложения в интернет напрямую.
3. Привяжите контейнер к `127.0.0.1`, наружу открывайте только 443.
4. Регулярно бэкапьте файл БД.
5. Rate-limiting на `/admin/login` и `/auth/*` (брутфорс) — через плагин proxy или middleware.
6. При росте нагрузки рассмотрите миграцию на PostgreSQL.
